## 一、项目概述类问题

### Q1：你们为什么选择做Flappy Bird这个游戏？

**回答**：

我们选择Flappy Bird有几个原因：

1. **经典且规则简单**：Flappy Bird规则简单，就是点击控制小鸟躲避障碍物，适合我们大一学生的水平
2. **涉及知识点全面**：虽然规则简单，但涉及面向对象、图形界面、碰撞检测、定时器等多个知识点，能全面锻炼我们的编程能力
3. **参考资源丰富**：网上有很多EasyX的教程和Flappy Bird的实现思路，方便我们学习参考
4. **可扩展性强**：基础版本完成后，我们可以不断添加新功能，比如道具系统、商店系统等

---

### Q2：你们的项目实现了哪些功能？

**回答**：

我们实现了以下功能：

- **核心玩法**：点击/空格控制小鸟飞行，躲避柱子障碍物
- **道具系统**：护盾、金币、二段跳、穿墙、缩小、慢动作、双倍得分共7种道具
- **商店系统**：用金币购买3种永久升级（开局护盾、Perfect+、DJ+）
- **评分系统**：基础得分、Perfect连击加成、Near Miss险过加成、Fever狂热模式
- **视觉增强**：开场动画、多背景切换、多样式柱子
- **音效系统**：拍翅膀、得分、碰撞、道具拾取等音效
- **数据持久化**：最高分和玩家存档保存到文件

## 二、技术设计类问题

### Q3：你们为什么采用面向对象的方式来设计？

**回答**：

我们采用面向对象设计是因为：

1. **模块化**：每个类负责一个功能，比如CBackGround管背景、CPlayerBird管小鸟，代码清晰易懂
2. **可维护性**：修改某个功能只需要改对应的类，不会影响其他部分
3. **可扩展性**：想加新功能只需要加新类，比如加道具系统就新建道具类
4. **学习目的**：课程要求我们学习面向对象编程，这个项目正好让我们实践类、继承、多态等概念

---

### Q4：你们的类是怎么设计的？为什么这样划分？

**回答**：

我们的类设计遵循"单一职责原则"：

| 类名 | 职责 |
|------|------|
| CBirdAPP | 主游戏类，负责整体逻辑和状态管理 |
| CPlayerBird | 小鸟类，负责小鸟的移动和显示 |
| CColumn/CColumnBox | 柱子类，负责障碍物的生成和管理 |
| CBackGround/CGround | 背景和地面类，负责场景显示 |
| CBeforeGame/CAfterGame | 界面类，负责开始和结束界面 |
| CGameFrame | 框架基类，负责Windows消息处理 |

这样划分让每个类只做一件事，代码清晰，也方便我们分工协作。

### Q5：你们是怎么实现碰撞检测的？

**回答**：

我们使用**矩形包围盒**（AABB）检测碰撞：

```cpp
bool CColumn::IfTouchBirdBox(int birdX, int birdY, int birdW, int birdH)
{
    // 检查水平方向是否重叠
    if (m_x <= birdX + birdW && m_x + IMG_COLUMN_WIDTH >= birdX)
    {
        // 检查垂直方向是否碰到柱子（不在缺口范围内）
        int gapTop = m_y + IMG_HALF_COLUMN_HEIGHT;
        int gapBottom = m_y + IMG_COLUMN_DOWN_Y;
        if (gapTop >= birdTop || gapBottom <= birdBottom)
            return true;
    }
    return false;
}
```

这种方法简单高效，适合我们初学者理解和实现。

## 三、困难与解决类问题

### Q6：你们遇到最大的困难是什么？是怎么解决的？

**回答**：

我们遇到最大的困难是**Windows消息机制的理解**。

**问题**：一开始我们用EasyX的`getmessage()`获取输入，但发现无法同时处理窗口关闭等系统消息，导致窗口无法正常关闭。

**解决过程**：

1. 查资料发现需要使用Win32的回调函数机制
2. 学习了`SetWindowLongPtr`设置窗口回调函数
3. 设计了消息映射表，将不同消息分发到不同处理函数
4. 参考了老师的框架代码，理解了消息循环的原理

**收获**：通过这个问题，我们深入理解了Windows消息机制，也学会了如何查资料解决问题。

---

### Q7：你们还遇到过哪些困难？

**回答**：

**困难1：定时器控制游戏节奏**

- 问题：一开始游戏速度不稳定，不同电脑上速度不一样
- 解决：学习使用Windows定时器`SetTimer`，用固定时间间隔控制游戏更新

**困难2：图片透明显示**

- 问题：小鸟图片有白色背景，直接显示会遮挡游戏画面
- 解决：学习使用`SRCPAINT`和`SRCAND`混合模式实现透明效果

**困难3：内存泄漏**

- 问题：柱子移出屏幕后没有释放内存
- 解决：学习使用`delete`释放内存，并用`nullptr`避免野指针

**困难4：中文显示乱码**

- 问题：EasyX默认不支持中文
- 解决：使用`_T()`宏和`settextstyle`设置中文字体

### Q8：你们是怎么调试程序的？

**回答**：

我们使用了多种调试方法：

1. **输出调试**：在关键位置用`cout`输出变量值，观察程序运行状态
2. **分段测试**：先测试单个功能，确认正确后再整合
3. **断点调试**：使用Visual Studio的断点功能，单步执行观察变量变化
4. **边界测试**：测试极端情况，比如分数为0、金币为0时会不会出错

调试过程中我们学会了**耐心和细心**，编程真的是"差一个字符都不行"。

## 五、反思总结类问题

### Q11：如果重新做这个项目，你们会怎么改进？

**回答**：

如果重新做，我们会：

1. **先做详细设计**：一开始我们直接写代码，后来发现设计不合理要重写。下次会先画类图、写设计文档
2. **统一代码规范**：一开始命名不统一，后来整理很麻烦
3. **更早使用Git**：一开始没用Git，代码丢失过一次，后来才学会用Git管理
4. **写更多注释**：有些代码现在自己都看不懂了
5. **先做最小可行版本**：一开始想做很多功能，结果基础功能都没做好

### Q13：你们觉得这个项目还有什么不足？

**回答**：

我们的项目还有很多不足：

1. **代码质量**：有些地方代码重复，可以进一步封装
2. **错误处理**：文件读写没有完善的错误处理
3. **性能优化**：每次创建柱子都加载图片，应该改为静态加载（我们已经优化了）
4. **用户体验**：没有暂停提示、没有音效开关
5. **测试不足**：没有写单元测试，主要靠手动测试

这些都是我们以后需要继续学习和改进的地方。

## 六、老师可能追问的问题

### Q14：解释一下你们的消息映射机制？

**回答**：

```cpp
// 消息映射表：将Windows消息ID映射到处理函数
map<UINT, Fun_Type> m_msgmap;

// 初始化时注册消息处理
INIT_MSGMAP(WM_LBUTTONDOWN, EM_MOUSE)  // 鼠标左键
INIT_MSGMAP(WM_KEYDOWN, EM_KEY)         // 键盘按下

// 回调函数中根据消息ID分发处理
if (P_G->m_msgmap.count(msgid)) {
    // 调用对应的处理函数
    (P_G->*(P_G->m_msgmap[msgid].p_fun_EM_MOUSE))(po);
}
```

这样设计的好处是**解耦**，新增消息只需要注册映射，不需要修改回调函数。

### Q15：你们的道具系统是怎么设计的？

**回答**：

道具系统设计：

1. **随机生成**：根据分数决定是否生成道具，每个分数只能生成一次
2. **类型随机**：生成时随机选择道具类型（护盾、金币、二段跳等）
3. **碰撞检测**：和柱子一样用矩形检测是否被小鸟碰到
4. **时效性**：每个道具有持续时间（ticks），每帧减少，归零时效果消失
5. **互斥性**：有些道具不能同时存在，比如已经有护盾时不再生成护盾
